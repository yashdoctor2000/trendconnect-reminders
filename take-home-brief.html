The scenario

Care teams need to track reminders for patients (for example "follow-up call" or "lab results due"). You'll build a small slice of this: an API to manage reminders and a simple UI to view and create them.

There's no real patient data here. Use made-up values. Treat the reminder's notes field as sensitive and handle it accordingly (see Requirements).

What to build

1. Backend (.NET Web API)

Model a Reminder with at least:

  • id (server-generated)
  • patientName (string, required)
  • title (string, required)
  • notes (string, optional; treat as sensitive)
  • dueDate (date/time, required)
  • status (Open | Done, defaults to Open)
  • recurrence (optional; None, Daily, or Weekly; defaults to None)
  • createdAt (server-generated)

Expose these endpoints:

  • GET /reminders: list reminders with pagination, filtering, and sorting (see below).
  • POST /reminders: create a reminder with validation.
  • GET /reminders/{id}: fetch a single reminder.
  • POST /reminders/{id}/complete: mark a reminder Done. If it recurs, this also creates the next occurrence (see below).
  • GET /reminders/{id}/history: return the audit trail of events for that reminder (see below).

Recurring reminders. When someone completes a Daily or Weekly reminder, the API should create the next occurrence automatically: a new Open reminder with its dueDate advanced by the interval, carrying over the other fields. A one-off (None) reminder just becomes Done. There are a few edge cases here, so think them through and write down your assumptions.

Domain events / audit trail. Every meaningful change (created, completed, next occurrence generated) should append an immutable event to the reminder's history instead of silently mutating state. GET /reminders/{id}/history returns those events in order, each with a type and a timestamp. We want to see append-only, event-style thinking, not a full event-sourcing framework.

Pagination, filtering, and sorting on GET /reminders:

  • Pagination: support page/pageSize (or a cursor), and return enough metadata for a client to page through (a total count or a next-page indicator).
  • Filtering: at least by status and a dueDate range.
  • Sorting: at least by dueDate and createdAt, ascending or descending.

Requirements:

  • Validation with clear, structured error responses and correct HTTP status codes (for example 400 for invalid input, 404 for missing).
  • Persistence: in-memory or SQLite is fine. No cloud or external infrastructure needed.
  • Safe logging: never log the notes field or any full record. Log identifiers only.
  • At least a few automated tests on the logic that matters most. The recurrence roll-over and the audit trail are good candidates.

2. Frontend (Angular)

  • A list view showing reminders (title, patient, due date, status, and whether it recurs), with a way to filter by status and page through results using the API's pagination.
  • A create form wired to POST /reminders, with basic client-side validation and visible loading and error states.
  • After a successful create, the new reminder should appear in the list.

Keep the styling minimal. Clean and readable is all we need. No design system required.

Ground rules & scope

Please do:

  • Keep it runnable with a documented one-command (or few-step) start for both API and UI.
  • Include a short README covering: how to run it, key decisions and tradeoffs, and what you'd do next with more time.
  • Use synthetic data only.

Please don't spend time on (explicitly out of scope):

  • Authentication / login, real databases, Docker, or deployment.
  • Pixel-perfect design or exhaustive test coverage.
  • Update/delete endpoints (unless you have time to spare and want to).

Using AI assistants (Copilot and the like) is fine. We use them too. But you own every line. In the live session we'll ask you to explain and change your code without an assistant, so make sure you understand all of it.

The live follow-up (~45 min)

After you submit, we'll meet for a short session where you'll:

  • Walk us through your solution, including the design and a couple of tradeoffs you made.
  • Extend it live. We'll ask you to add a small enhancement to the code you already wrote. Nothing exotic. If you understand your own solution, you'll be fine.

Have your dev environment ready to run and edit the project during the call.

How we evaluate

Here's what we pay attention to. A strong showing in most of these matters more than perfection in any one:

  • Correctness: it works, meets the core requirements, and handles sensible edge cases.
  • Code quality and structure: clean, idiomatic, with concerns separated well across the API and UI.
  • Testing: meaningful tests on the logic that matters.
  • Data handling: sensible validation, safe logging, and safe error responses.
  • Integration and UX: the frontend and backend work together cleanly, with clear loading and error states.
  • Communication: your README, and how well you explain and extend the work live.

You don't need to cover every edge case, add fancy styling, or build a big surface area. Show us clean, well-reasoned engineering.
